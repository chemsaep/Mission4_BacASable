== Mission 1 : Architecture et Modélisation

=== 1.1 Modèle Relationnel (MLD)
* *Client* (Id, raison sociale, adresse, telephone, #IdProgrammefidelité)
* *Point de vente* (id, nomPDV, adresse, #idClient, #idformule)
* *Formule* (id, libelle, prix, nbVendeursSimultanes)
* *ProgrammeFidelite* (id, libelleProg)
* *Langue* (id, libelleLangue)

=== 1.2 Processus d'inscription sécurisé
Le processus suit les étapes suivantes :
1. Entrée des informations.
2. Création d'un token temporaire.
3. Envoi par mail d'un lien d'activation.
4. L'utilisateur choisit son mot de passe (MDP).
5. Le MDP est haché avant stockage.

=== 1.3 Problématique de l'historique
La structure initiale de la table `FormuleSouscrite(idPointDeVente, idFormule)` pose deux problèmes majeurs pour l'historique de facturation, notamment des contraintes de **clés primaires** qui empêchent de stocker plusieurs formules successives pour un même point de vente.

=== 1.4 Nouvelle structure de table
[source,sql]
----
FormuleSouscrite (#idPointDeVente, dateDebut, #idFormule, dateFin)
----


=== 1.5 Automatisation de l'audit (Trigger)
[source,sql]
----
CREATE TRIGGER trigger  
AFTER UPDATE ON FormuleSouscrite  
FOR EACH ROW  
BEGIN  
    IF OLD.idFormule <> NEW.idFormule THEN  
        INSERT INTO AuditFormule (idPDV, ancienneFormule, nouvelleFormule, dateModif)  
        VALUES (OLD.idPointDeVente, OLD.idFormule, NEW.idFormule, NOW());  
    END IF;  
END;
----

=== 2.3 Sécurisation avec PreparedStatement
[source,java]
----
String requete = "SELECT c.nom, c.prenom, c.tel, c.mail "  
               + "FROM Conso c JOIN Vente v ON c.id = v.idConso "  
               + "WHERE c.id NOT IN (SELECT id FROM ConsoFidele) "  
               + "AND v.dateVente BETWEEN ? AND ? "  
               + "GROUP BY c.id, c.nom, c.prenom, c.tel, c.mail "  
               + "HAVING COUNT(v.id) > ?";

PreparedStatement pstmt = dbConnect.prepareStatement(requete);
pstmt.setDate(1, java.sql.Date.valueOf(dateDeb));  
pstmt.setDate(2, java.sql.Date.valueOf(dateFin));  
pstmt.setInt(3, seuilVentes);

ResultSet res = pstmt.executeQuery();

== Questions Finales

1. **Norme NF525 :** Oblige le logiciel à garantir l'inaltérabilité, la sécurité (archivage) et la traçabilité des données pour empêcher la fraude fiscale.
2. **Danger du MDP par mail :** Le message peut être intercepté par un tiers, offrant un accès immédiat sans effort de piratage.
3. **Absence de dates :** Sans `dateDebut` ou `dateFin`, la table ne stocke que l'état actuel, rendant impossible la facturation rétroactive exacte.
4. **Avantage du Trigger :** Il s'exécute automatiquement côté base de données, garantissant l'audit même si l'application Java oublie de le faire.
5. **WHERE vs HAVING :** `WHERE` filtre avant le regroupement ; `HAVING` filtre après le `GROUP BY`.
6. **Exemple d'Injection SQL :** L'utilisation de `' OR '1'='1` dans un champ de login pour court-circuiter la vérification du mot de passe.
